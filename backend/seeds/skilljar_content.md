# Skilljar Course Content -- CCA Exam Reference

## Course: Building with the Claude API (85 lessons)

### Section: Accessing Claude with the API

### Lesson: Accessing the API
- Five-step request flow: request to server, request to Anthropic API, model processing, response to server, response to client
- Anti-pattern: Never make requests to the Anthropic API directly from client-side code (exposes API key)
- Best practice: Web/mobile app sends requests to YOUR server, which communicates with Anthropic API
- Anthropic provides SDKs for: Python, TypeScript, JavaScript, Go, and Ruby
- Required request fields: API Key, Model name, Messages list, Max Tokens limit
- Claude processing stages: tokenization, embedding, contextualization, generation
- Stop conditions: max tokens reached, end-of-sequence token (natural ending), stop sequence encountered
- API response contains: Message (text), Usage (input/output token counts), Stop Reason

### Lesson: Making a request
- Core function: client.messages.create() requires model, max_tokens, and messages
- max_tokens is a SAFETY MECHANISM, not a target -- Claude stops if it hits the maximum but doesn't try to reach it
- Two message types: "user" (human input) and "assistant" (Claude responses)
- Each message is a dictionary with "role" and "content" fields
- Extract response text: message.content[0].text
- Best practice: Store API key in .env file, add .env to .gitignore
- Default model example: "claude-sonnet-4-0"

### Lesson: Multi-Turn conversations
- Key fact: Claude does NOT store conversation history -- each request is completely independent
- To maintain context: manually maintain a list of all messages AND send the complete message history with every request
- Flow: send user message -> add Claude's response as assistant message -> add follow-up as user message -> send entire history
- Anti-pattern: Sending only the latest message without conversation history (Claude won't know context)

### Lesson: System prompts
- System prompts customize Claude's tone, style, and approach
- Passed via the system= parameter in the create function call
- Important: Claude's API doesn't accept system=None -- conditionally include the system parameter only when provided
- Best practice: Make chat functions accept system prompts as optional parameters
- Use case example: Math tutor that gives hints rather than direct answers

### Lesson: Temperature
- Temperature is a decimal value between 0 and 1
- Low temperature (0.0-0.3): factual responses, coding assistance, data extraction, content moderation
- Medium temperature (0.4-0.7): summarization, educational content, problem-solving, creative writing with constraints
- High temperature (0.8-1.0): brainstorming, creative writing, marketing content, joke generation
- At temperature 0.0: completely deterministic (highest probability token always chosen)
- At temperature 1.0: probabilities spread more evenly, more varied/creative outputs
- Temperature doesn't GUARANTEE different outputs -- just changes the probability

### Lesson: Response streaming
- Without streaming, responses can take 10-30 seconds
- Stream event types: MessageStart, ContentBlockStart, ContentBlockDelta, ContentBlockStop, MessageDelta, MessageStop
- ContentBlockDelta events contain the actual generated text
- Enable streaming: stream=True in messages.create()
- Simplified streaming: client.messages.stream() with stream.text_stream
- Get complete message after streaming: stream.get_final_message()
- Best practice: Use streaming for UX, get_final_message() for storage

### Lesson: Structured data
- Problem: Claude adds explanatory text around structured data (JSON, code, etc.)
- Solution: Assistant Message Prefilling + Stop Sequences
- Technique: Prefill assistant message with "```json", set stop_sequences=["```"]
- Claude thinks it already started a code block, writes just the content, stops at closing marker
- Works for: JSON, Python code, bulleted lists, CSV data

### Section: Prompt Evaluation

### Lesson: Prompt evaluation
- Prompt engineering = techniques for writing better prompts
- Prompt evaluation = measuring effectiveness through automated testing
- Three options after writing a prompt: (1) test once (risky), (2) test a few times (better), (3) run through eval pipeline (best)
- Anti-pattern: Testing a prompt only once or a few times before production
- Best practice: Run prompts through evaluation pipeline for objective metrics

### Lesson: A typical eval workflow
- Five steps: Draft prompt -> Create eval dataset -> Feed through Claude -> Feed through grader -> Change prompt and repeat
- Grading scale: typically 1-10 where 10 is perfect
- Average score across test cases gives objective measurement
- Dataset can be hand-assembled or generated by Claude
- Best practice: Compare different prompt versions numerically

### Lesson: Generating test datasets
- Use faster models (like Haiku) for generating test data
- Use prefilling and stop sequences to get clean JSON output from generation
- Save datasets to JSON files for reuse
- Best practice: Generate diverse test cases covering different output types

### Lesson: Running the eval
- Core pipeline: run_eval -> run_test_case -> run_prompt
- Each result contains: output, test_case, score
- Even with Claude Haiku, processing a full dataset can take ~30 seconds
- The pipeline is the foundation; complexity comes in prompts, grading, and optimization

### Lesson: Model based grading
- Three grader types: Code graders, Model graders, Human graders
- Code graders: output length, word presence, syntax validation, readability scores
- Model graders: response quality, instruction following, completeness, helpfulness, safety
- Human graders: most flexible but time-consuming
- Best practice: Ask for strengths, weaknesses, and reasoning alongside the score (models default to middling ~6 without this)
- Evaluation criteria example: Format, Valid Syntax, Task Following

### Lesson: Code based grading
- Syntax validation functions return 10 (valid) or 0 (invalid)
- json.loads() for JSON, ast.parse() for Python, re.compile() for Regex
- Dataset needs "format" field to route to correct validator
- Combine model grader + code grader scores (e.g., average)

### Section: Prompt Engineering Techniques

### Lesson: Prompt engineering
- Iterative process: Set goal -> Write prompt -> Evaluate -> Apply techniques -> Re-evaluate
- Use PromptEvaluator class with max_concurrent_tasks parameter
- Start with low concurrency (3) to avoid rate limit errors
- Keep test cases low (2-3) during development for faster iteration
- Best practice: Make one change at a time, evaluate impact

### Lesson: Being clear and direct
- First line of prompt is the most important part
- Use simple, unambiguous language
- Use instructions, not questions (e.g., "Write..." not "Could you...")
- Start with direct action verbs: Write, Create, Generate, Identify
- Example improvement: "What should this person eat?" -> "Generate a one-day meal plan for an athlete that meets their dietary restrictions"

### Lesson: Being specific
- Two types of guidelines: Output Quality Guidelines and Process Steps
- Output quality guidelines: length, structure, format, attributes, tone
- Process steps: specific sequential steps for Claude to follow
- Use output guidelines in almost every prompt
- Use process steps for complex problems, troubleshooting, decision-making, critical thinking
- In testing, adding guidelines improved score from 3.92 to 7.86

### Lesson: Structure with XML tags
- XML tags create clear boundaries in prompts with lots of data
- Use custom, descriptive tag names (e.g., <sales_records>, <athlete_information>, <my_code>)
- Most useful when: including large context, mixing content types, complex prompts with multiple variables
- Example: <athlete_information>...</athlete_information>

### Lesson: Providing examples
- One-shot: single example to establish pattern
- Multi-shot: multiple examples covering different scenarios
- Use examples for: corner cases, complex output formats, exact style/tone, ambiguous inputs
- Wrap examples in XML tags: <sample_input>, <ideal_output>
- Best practice: Use highest-scoring eval outputs as examples
- Add context explaining WHY the output is good
- Examples show rather than tell -- more reliable than describing requirements

### Section: Tool Use with Claude

### Lesson: Introducing tool use
- Tools allow Claude to access information from the outside world
- Tool use flow: Initial request with tool instructions -> Claude requests tool -> Server fetches data -> Data sent back to Claude -> Final response
- Key benefits: real-time information, external system integration, dynamic responses, structured interaction

### Lesson: Tool functions
- Tool functions are plain Python functions executed when Claude needs extra information
- Best practices: descriptive names, validate inputs, provide meaningful error messages
- Claude can see error messages and may retry with corrected parameters

### Lesson: Tool schemas
- JSON Schema describes function parameters for Claude
- Three main parts: name, description, input_schema
- Tool description best practices: 3-4 sentences, describe when to use, explain return data, detail each argument
- Can use Claude itself to generate schemas
- Naming pattern: function_name + function_name_schema
- Use ToolParam type from anthropic.types for type safety

### Lesson: Sending tool results
- Tool result block properties: tool_use_id (must match), content (string), is_error (boolean)
- Claude can request multiple tool calls in a single response -- each gets unique ID
- Must include complete conversation history + tool schema in follow-up request
- Always include tool schema even if not expecting another tool call

### Lesson: Multi-turn conversations with tools
- Check response.stop_reason == "tool_use" to detect if Claude wants to use a tool
- Loop until Claude stops requesting tools
- Handle multiple tool calls per response: filter for tool_use blocks
- Error handling: always provide a result block, even on error (set is_error=True)
- Scalable routing: map tool names to implementations in run_tool function

### Lesson: Implementing multiple turns
- Conversation loop: send message -> check stop_reason -> run tools -> send results -> repeat
- Claude can request multiple tools in single response
- Each tool_use_id must match corresponding tool_result_id
- Always handle errors gracefully -- provide is_error=True result block

### Lesson: Using multiple tools
- Pattern for adding tools: (1) Create function, (2) Define schema, (3) Add to tools list, (4) Add case in run_tool
- Pass all tool schemas in the tools array
- Claude determines which tools to call and in what order

### Lesson: Fine grained tool calling
- With streaming + tools: InputJsonEvent contains partial_json and snapshot
- API buffers chunks and validates complete top-level key-value pairs before sending
- Fine-grained tool calling (fine_grained=True): disables JSON validation, get chunks immediately
- Trade-off: faster streaming but YOU must handle invalid JSON
- Use when: showing real-time progress, starting partial processing early

### Lesson: The text edit tool
- Built-in tool: Claude knows the schema but YOU must provide the implementation
- Capabilities: view files/directories, view line ranges, replace text, create files, insert text, undo edits
- Schema versions vary by model (text_editor_20250124 for claude-3-7-sonnet, text_editor_20241022 for claude-3-5-sonnet)
- Include small schema stub in requests; Claude expands it internally

### Lesson: The web search tool
- Organization must enable Web Search in settings console first
- Schema: type="web_search_20250305", name="web_search", max_uses (limits searches per request)
- Response blocks: Text, ServerToolUseBlock, WebSearchToolResultBlock, WebSearchResultBlock, Citation blocks
- Restrict domains with allowed_domains field (e.g., ["nih.gov"] for medical info)
- Claude automatically decides when web search would help answer questions

### Section: RAG and Agentic Search

### Lesson: BM25 lexical search
- Semantic search alone misses exact term matches
- Hybrid search: run semantic + lexical (BM25) in parallel, merge results
- BM25 algorithm: tokenize query -> count term frequency -> weight by importance (rare terms score higher) -> find best matches
- BM25 excels at: exact matches, technical terms, IDs, specific phrases
- Complementary strengths: semantic understands context, lexical finds exact matches

### Lesson: Text embeddings
- Text embedding = numerical representation of meaning in text
- Each number ranges from -1 to +1, represents some quality/feature
- We don't know precisely what each dimension represents
- Recommended embedding provider: VoyageAI (Anthropic doesn't provide embeddings)
- Model: voyage-3-large

### Section: Features of Claude

### Lesson: Extended thinking
- Extended thinking = Claude's "scratch paper" for complex reasoning
- Response contains two parts: thinking process + final answer
- Benefits: better reasoning, increased accuracy, transparency
- Trade-offs: higher costs (pay for thinking tokens), increased latency
- When to use: after optimizing prompts, when standard prompting isn't meeting accuracy requirements
- Thinking budget minimum: 1024 tokens; max_tokens must be greater than thinking budget
- Signature system: cryptographic token prevents tampering with thinking text
- Redacted thinking: occurs when safety systems flag thinking process
- Not compatible with: message pre-filling and temperature (see docs for full list)

### Lesson: Image support
- Up to 100 images per request
- Max 5MB per image
- Single image: max 8000px height/width
- Multiple images: max 2000px height/width
- Image formats: base64 encoding or URL
- Token calculation: (width px x height px) / 750
- Same prompting techniques apply to images (guidelines, one-shot examples, step-by-step)

### Lesson: Prompt caching
- Caches preprocessing work (tokenization, embeddings, context analysis) instead of discarding it
- Benefits: faster responses, lower costs, automatic optimization
- Limitations: cache lives for only ONE HOUR, only beneficial for repeated content
- Best for: document analysis workflows, iterative editing with constant base content

### Lesson: Code execution and the Files API
- Files API: upload files ahead of time, reference by file ID later
- Code execution: server-based tool, runs Python in isolated Docker container
- Docker container has NO network access
- Combine them: upload data via Files API, reference in code execution
- Container upload block: {"type": "container_upload", "file_id": file_metadata.id}
- Code execution schema: {"type": "code_execution_20250522", "name": "code_execution"}
- Claude can execute code multiple times during single conversation
- Download generated files using Files API

### Section: Model Context Protocol (in Building with the Claude API)

### Lesson: MCP clients
- MCP client = communication bridge between your server and MCP servers
- Transport agnostic: standard I/O (same machine), HTTP, WebSockets
- Message types: ListToolsRequest/Result (discover tools), CallToolRequest/Result (execute tools)
- Complete flow: user query -> server asks MCP client for tools -> client asks MCP server -> Claude decides tool -> MCP executes -> results flow back

### Lesson: Defining resources (MCP)
- Resources expose data (similar to GET endpoints)
- Two types: Direct Resources (static URIs like docs://documents) and Templated Resources (parameterized like docs://documents/{doc_id})
- Decorator: @mcp.resource("uri", mime_type="type")
- SDK auto-serializes return values and auto-parses template parameters
- Test with MCP Inspector: uv run mcp dev mcp_server.py

### Section: Anthropic Apps

### Lesson: Claude Code in action
- Claude Code uses generic tools: bash, read, write, edit, glob, grep
- Does NOT have specialized tools like "refactor code" or "install dependencies"
- Instead, Claude figures out how to use basic tools to accomplish complex tasks

### Section: Agents and Workflows

### Lesson: Agents and workflows
- Workflows: predefined series of calls, for known problems, exact step sequences
- Agents: Claude gets tools and figures out how to complete goals
- Use workflows when you can picture the exact flow
- Use agents when you don't know exact tasks/parameters
- Evaluator-Optimizer pattern: Producer creates output -> Grader evaluates -> Feedback loop -> Iterate

### Lesson: Parallelization workflows
- Split single task into multiple parallel sub-tasks
- Each sub-task has specialized prompt/criteria
- Aggregate results in final step
- Benefits: focused attention, easier per-task optimization, better scalability, improved reliability
- Use when: complex decisions with independent evaluations

### Lesson: Chaining workflows
- Break large task into smaller, sequential subtasks
- Each step focuses on one aspect
- Key benefit: FOCUS -- Claude does one task well vs. juggling multiple
- Long prompt problem: Claude may ignore constraints in massive prompts
- Solution: two-step chain -- generate content, then revise for constraint compliance

### Lesson: Routing workflows
- Categorize incoming requests, route to specialized pipelines
- Two-step process: (1) Claude categorizes input, (2) Use category to select prompt template
- User input goes to ONE specialized pipeline, not all
- Best for: diverse request types needing different handling

### Lesson: Agents and tools
- Agents shine when you don't know the exact steps needed
- Tools should be reasonably ABSTRACT, not hyper-specialized
- Claude Code example: bash, read, write, edit, glob, grep (generic tools)
- Best practice: provide combinable tools Claude can chain creatively
- Agents can ask users for additional input when needed

### Lesson: Workflows vs agents
- Workflows: higher accuracy, easier to test, more predictable, solve specific problems
- Agents: more flexible UX, handle novel situations, creative tool combination
- Workflow downsides: less flexible, constrained UX, more upfront planning
- Agent downsides: lower success rate, harder to test/evaluate, less predictable
- Best practice: ALWAYS focus on implementing workflows where possible, only resort to agents when truly required
- Users want products that work consistently -- reliability over fancy agents

## Course: Claude Code in Action (21 lessons)

### Lesson: What is a coding assistant?
- Coding assistants use language models + tool use to tackle programming tasks
- Process: Gather context -> Formulate plan -> Take action
- Language models alone can only process/return text -- they need TOOLS to interact with the world
- Tool use flow: user asks -> coding assistant adds tool instructions -> model requests tool -> assistant executes -> model gets results
- Claude models (Opus, Sonnet, Haiku) are particularly strong at tool use
- Claude Code navigates codebases WITHOUT requiring indexing (better security)

### Lesson: Adding context
- /init command: analyzes codebase, creates CLAUDE.md summary file
- CLAUDE.md: guides Claude through codebase, included in every request (persistent system prompt)
- Three CLAUDE.md locations:
  - CLAUDE.md: committed to source control, shared with team
  - CLAUDE.local.md: personal, NOT shared with team
  - ~/.claude/CLAUDE.md: used with ALL projects on your machine
- # command enters "memory mode" to edit CLAUDE.md files
- @ symbol: mention specific files to include their contents in request
- @ in CLAUDE.md: auto-includes file contents in every request

### Lesson: Making changes
- Ctrl+V (not Cmd+V) to paste screenshots into Claude
- Planning Mode (Shift+Tab twice): thorough codebase exploration before implementing
- Thinking modes: "Think", "Think more", "Think a lot", "Think longer", "Ultrathink"
- Each thinking mode gives progressively more tokens for deeper analysis
- Planning Mode = broad codebase understanding; Thinking Mode = deep logic problems

### Lesson: Controlling context
- Escape: stops Claude mid-response to redirect
- Escape + # shortcut: fix repetitive errors by adding memories
- Double-tap Escape: rewind conversation to earlier point
- /compact: summarizes conversation while preserving key knowledge
- /clear: completely removes conversation history (fresh start)
- Use /compact when continuing related tasks; /clear when switching to unrelated task

### Lesson: Custom commands
- Create .claude/commands/ directory with markdown files
- Filename becomes command name (e.g., audit.md creates /audit)
- $ARGUMENTS placeholder for dynamic input
- Must restart Claude Code for new commands to be recognized
- Benefits: automation, consistency, context, flexibility

### Lesson: MCP servers with Claude Code
- Install MCP server: claude mcp add playwright npx @playwright/mcp@latest
- Pre-approve permissions in .claude/settings.local.json: "allow": ["mcp__playwright"]
- Note double underscores: mcp__playwright
- Playwright MCP gives Claude ability to control a web browser
- Claude can see actual visual output (not just code) for informed decisions

### Lesson: Github integration
- /install-github-app: sets up official GitHub integration
- Two workflows: @claude mentions (issues/PRs) and automatic PR reviews
- Customize with: project setup steps, custom instructions, MCP config, tool permissions
- In GitHub Actions: must explicitly list ALL allowed tools (no shortcuts)

### Lesson: Introducing hooks
- PreToolUse hooks: run BEFORE a tool is called (can block operations)
- PostToolUse hooks: run AFTER a tool is called (can't block, but can enhance)
- Hook locations: global (~/.claude/settings.json), project (.claude/settings.json), local (.claude/settings.local.json)
- /hooks command to create hooks interactively
- Practical uses: code formatting, testing, access control, code quality, logging, validation

### Lesson: Defining hooks
- Hook receives JSON data via stdin: session_id, tool_name, tool_input
- Exit code 0: allow operation to proceed
- Exit code 2: BLOCK the tool call (PreToolUse only)
- Stderr output sent to Claude as feedback when blocking
- Matcher uses pipe (|) as OR operator: "Read|Grep"

### Lesson: Implementing a hook
- Example: blocking .env file access with PreToolUse hook on Read|Grep
- Hook reads JSON from stdin, checks file_path for .env, exits with code 2 to block
- Restart Claude Code after changing hooks
- Claude understands when operations are blocked by hooks

### Lesson: Useful hooks
- TypeScript type checking hook: runs tsc --noEmit after every file edit, feeds errors back to Claude
- Query duplication hook: launches separate Claude instance to review for duplicate queries
- Can use Claude's TypeScript SDK to programmatically interact with AI in hooks

### Lesson: The Claude Code SDK
- Available for TypeScript, Python, and CLI
- Runs exact same Claude Code as terminal version
- Default: read-only permissions (Read, Grep, Glob only)
- Enable write: add allowedTools: ["Edit"] in query options
- Inherits settings from Claude Code instances in same directory
- Use cases: git hooks, build scripts, CI/CD pipelines, automated code review

## Course: Introduction to Agent Skills (6 lessons)

### Lesson: What are skills?
- Skills = folders of markdown instructions that Claude Code discovers and uses automatically
- Each skill has a SKILL.md file with name and description in frontmatter
- Claude uses DESCRIPTION to match skills to requests (semantic matching)
- Personal skills: ~/.claude/skills (follow you across all projects)
- Project skills: .claude/skills (shared via version control)
- Skills load ON DEMAND (unlike CLAUDE.md which loads every conversation)
- Skills activate AUTOMATICALLY (unlike slash commands which require explicit invocation)
- Rule of thumb: if you repeat the same instructions to Claude, that's a skill

### Lesson: Creating your first skill
- Skill = directory containing SKILL.md with frontmatter (name, description) + instructions
- Claude loads only names and descriptions at startup, matches incoming requests
- Confirmation prompt before loading full skill content into context
- Priority hierarchy: Enterprise > Personal > Project > Plugins
- Restart Claude Code after creating/updating/removing skills
- Use descriptive names to avoid conflicts (e.g., "frontend-review" not just "review")

### Lesson: Configuration and multi-file skills
- Required metadata: name, description
- Optional metadata: allowed-tools, model
- allowed-tools restricts what Claude can do when skill is active (e.g., Read, Grep, Glob, Bash for read-only)
- Progressive disclosure: keep SKILL.md under 500 lines, link to supporting files
- Skill directory structure: scripts/, references/, assets/
- Scripts execute without loading contents into context (only output consumes tokens)
- Description should answer: What does skill do? When should Claude use it?
- name: lowercase letters, numbers, hyphens only, max 64 chars
- description: max 1,024 characters

### Lesson: Skills vs. other Claude Code features
- CLAUDE.md: always-on project standards (loads every conversation)
- Skills: on-demand task-specific expertise (loads when matched)
- Subagents: isolated execution contexts (separate context)
- Hooks: event-driven (fire on file saves, tool calls)
- MCP servers: external tools and integrations
- Skills are request-driven; hooks are event-driven
- Combine features rather than forcing everything into one approach

### Lesson: Sharing skills
- Project skills: commit to .claude/skills, shared via Git
- Plugins: distribute across repositories via marketplaces
- Enterprise managed settings: organization-wide, highest priority
- IMPORTANT: Subagents do NOT automatically see skills
- Must explicitly list skills in custom agent's frontmatter skills field
- Built-in agents (Explorer, Plan, Verify) CANNOT access skills at all
- Only custom subagents in .claude/agents can use skills

### Lesson: Troubleshooting skills
- Skills validator tool: catches structural problems first
- Skill doesn't trigger: improve description, add trigger phrases matching actual usage
- Skill doesn't load: SKILL.md must be inside named directory, filename exactly "SKILL.md"
- Wrong skill used: descriptions too similar, make them distinct
- Priority conflicts: enterprise skills override personal with same name
- Runtime errors: check dependencies, file permissions (chmod +x), path separators (forward slashes everywhere)
- Use claude --debug to see loading errors

## Key Facts Summary for CCA Exam Questions

### API Fundamentals
- Never expose API keys in client-side code
- Required fields: model, max_tokens, messages
- max_tokens is a safety limit, NOT a target
- Claude is stateless -- send full conversation history every request
- System prompts: use system= parameter, don't pass None
- Temperature range: 0 to 1 (0=deterministic, 1=creative)
- Stop reasons: max_tokens, end_turn, stop_sequence
- SDKs available for: Python, TypeScript, JavaScript, Go, Ruby

### Prompt Engineering
- Be clear and direct; use action verbs (Write, Create, Generate)
- Be specific: output quality guidelines + process steps
- Use XML tags for structure in complex prompts
- Provide examples (one-shot/multi-shot) for edge cases
- Assistant message prefilling + stop sequences for structured output

### Tool Use
- Tool schema has: name, description, input_schema
- stop_reason == "tool_use" indicates Claude wants to call a tool
- tool_result must include matching tool_use_id
- Claude can request multiple tools in one response
- Text editor tool: Claude knows schema, you implement functions
- Web search tool: needs org-level enable, max_uses controls searches
- Fine-grained tool calling: disables API-side JSON validation
- Best practice for tool functions: validate inputs, provide meaningful error messages
- ToolParam type from anthropic.types for type safety

### Features
- Extended thinking: min budget 1024 tokens, incompatible with prefilling and temperature
- Extended thinking includes cryptographic signature to prevent tampering
- Redacted thinking: occurs when safety systems flag thinking process
- Image limits: 100 per request, 5MB each, token cost = (w*h)/750
- Single image max: 8000px; multiple images max: 2000px
- Prompt caching: cache lives 1 hour, saves preprocessing work
- Code execution: isolated Docker, no network access, combine with Files API
- Code execution schema: type="code_execution_20250522"
- Embeddings: use VoyageAI (not Anthropic), model voyage-3-large

### Architecture Patterns
- Workflows: predefined steps, high reliability, use when flow is known
- Agents: flexible, goal + tools, lower reliability
- Parallelization: split into parallel sub-tasks, aggregate results
- Chaining: sequential focused subtasks
- Routing: categorize then route to specialized pipeline
- Evaluator-Optimizer: produce -> grade -> feedback -> iterate
- Best practice: PREFER WORKFLOWS over agents for reliability
- Tools should be reasonably ABSTRACT, not hyper-specialized

### MCP (Model Context Protocol)
- Three primitives: Tools, Resources, Prompts
- Transport agnostic: stdio, HTTP, WebSockets
- MCP Inspector for testing servers: uv run mcp dev server.py
- Resources: Direct (static URI) vs Templated (parameterized URI)
- Install MCP in Claude Code: claude mcp add <name> <command>
- Permission pre-approval: mcp__<servername> in settings allow list (double underscores)

### Claude Code
- Built-in tools: bash, read, write, edit, glob, grep (generic, not specialized)
- CLAUDE.md locations: CLAUDE.md (shared), CLAUDE.local.md (personal), ~/.claude/CLAUDE.md (global)
- /init creates CLAUDE.md; # enters memory mode; @ mentions files
- Planning Mode: Shift+Tab twice for thorough exploration before implementation
- Thinking modes: Think, Think more, Think a lot, Think longer, Ultrathink
- Context control: Escape (stop), double Escape (rewind), /compact (summarize), /clear (reset)
- Custom commands: .claude/commands/<name>.md, $ARGUMENTS placeholder
- Hooks: PreToolUse (can block, exit code 2) and PostToolUse (enhance only)
- Hook exit codes: 0 = allow, 2 = block (PreToolUse only)
- Claude Code SDK: read-only by default, available for TypeScript/Python/CLI
- GitHub integration: /install-github-app, @claude mentions, auto PR reviews

### Agent Skills
- Skills = SKILL.md files with frontmatter (name, description) that Claude discovers automatically
- Personal skills: ~/.claude/skills; Project skills: .claude/skills
- Priority: Enterprise > Personal > Project > Plugins
- Skills load on demand (unlike CLAUDE.md which loads every time)
- Skills activate automatically (unlike slash commands which need explicit invocation)
- Required frontmatter: name (max 64 chars), description (max 1024 chars)
- Optional frontmatter: allowed-tools, model
- Keep SKILL.md under 500 lines; use progressive disclosure with supporting files
- Subagents do NOT automatically see skills -- must explicitly list in agent frontmatter
- Built-in agents (Explorer, Plan, Verify) CANNOT access skills
- Troubleshooting: use skills validator, claude --debug for loading errors

### Eval Best Practices
- Three grader types: code, model, human
- Always ask for reasoning alongside scores (prevents middling ~6 defaults)
- Use Haiku for generating test data (faster/cheaper)
- Keep test cases low during development (2-3)
- Make one change at a time, evaluate impact
- Syntax validation: json.loads() for JSON, ast.parse() for Python, re.compile() for Regex
