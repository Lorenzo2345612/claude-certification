Anti-patrones del Loop Agentico (pregunta frecuente del examen)

  • Parsear texto en lenguaje natural para determinar si el loop debe terminar
  • Usar caps arbitrarios de iteracion como mecanismo primario de parada

Correcto: Continuar cuando stop_reason == "tool_use" y terminar cuando stop_reason == "end_turn"

# Agentic loop correcto - patron del examen import anthropic client = anthropic.Anthropic() messages = [{"role": "user", "content": prompt}] while True: response = client.messages.create( model="claude-opus-4-6", max_tokens=4096, tools=tools, messages=messages ) # Agregar respuesta al historial messages.append({"role": "assistant", "content": response.content}) if response.stop_reason == "end_turn": break # Loop terminado correctamente # Ejecutar herramientas y agregar resultados tool_results = [] for block in response.content: if block.type == "tool_use": result = execute_tool(block.name, block.input) tool_results.append({ "type": "tool_result", "tool_use_id": block.id, "content": result }) messages.append({"role": "user", "content": tool_results})

1.2 Patron Coordinator-Subagent

Arquitectura hub-and-spoke donde un coordinador gestiona toda la comunicacion entre sub-agentes.

  • Sub-agentes tienen contexto aislado — NO heredan el historial del coordinador
  • El coordinador analiza requerimientos y selecciona dinamicamente que sub-agentes invocar
  • Toda comunicacion pasa por el coordinador para observabilidad
  • Loops de refinamiento iterativo: evaluar sintesis, re-delegar si hay gaps

Riesgo comun de examen

La descomposicion demasiado estrecha del coordinador lleva a cobertura incompleta. Ejemplo: "AI en industrias creativas" descompuesto solo en artes visuales, omitiendo musica, escritura y cine.

1.3 Spawning de Subagentes

El Task tool (o Agent tool en Agent SDK) es el mecanismo para spawnear subagentes.

  • allowedTools debe incluir "Task" para que el coordinador pueda invocar subagentes
  • Contexto debe pasarse explicitamente en el prompt — no hay herencia automatica
  • Usar datos estructurados para separar contenido de metadata (URLs, paginas)
  • Ejecucion paralela: emitir multiples Task calls en una sola respuesta

1.4 Enforcement y Handoffs

Diferencia critica entre enforcement programatico (hooks/gates) y prompt-based guidance.

  • Cuando compliance determinista es requerido, prompt instructions solas tienen tasa de fallo no-zero
  • Prerequisitos programaticos bloquean tools downstream hasta completar pasos previos
  • Handoff estructurado: customer ID + root cause + monto + accion recomendada

Regla clave para el examen

Enforcement programatico > Prompt instructions cuando el error tiene consecuencias financieras o de seguridad. Ejemplo: bloquear process_refund hasta que get_customer retorne un ID verificado.

1.5 Agent SDK Hooks

Hooks interceptan tool calls y tool results para transformacion y enforcement.

  • PostToolUse: normalizar formatos heterogeneos (timestamps, codigos de status)
  • Tool call interception: bloquear acciones que violan politicas (refunds > $500)
  • Hooks para garantias deterministas vs prompts para compliance probabilistico
# Hook PostToolUse - normalizar datos hooks = { "PostToolUse": [ HookMatcher( matcher="get_customer|lookup_order", hooks=[normalize_timestamps] ) ] } # Hook de interception - bloquear refunds altos async def block_high_refunds(input_data, tool_use_id, ctx): amount = input_data["tool_input"].get("amount", 0) if amount > 500: return {"decision": "block", "message": "Escalate to human"}

1.6 Estrategias de Descomposicion

Elegir entre prompt chaining (pipelines fijos) y descomposicion adaptativa.

  • Prompt chaining: reviews multi-aspecto predecibles (per-file + cross-file pass)
  • Descomposicion dinamica: tareas open-ended donde subtasks dependen de hallazgos previos
  • Dividir code reviews grandes en analisis local per-file + pass de integracion cross-file

1.7 Session State y Forking

Gestion de sesiones para continuidad y exploracion paralela.

  • --resume <session-name> para continuar conversaciones especificas
  • fork_session para crear ramas independientes desde un baseline compartido
  • Nueva sesion con resumen estructurado es mas confiable que resumir con tool results obsoletos
  • Informar al agente sobre cambios en archivos al resumir sesiones
Respuesta Exacta de la API con tool_use
{ "id": "msg_01Aq9w938a90dw8q", "model": "claude-opus-4-6", "stop_reason": "tool_use", "role": "assistant", "content": [ { "type": "text", "text": "I'll check the current weather in San Francisco for you." }, { "type": "tool_use", "id": "toolu_01A09q90qw90lq917835lq9", "name": "get_weather", "input": { "location": "San Francisco, CA", "unit": "celsius" } } ] }

Detalle clave

Claude puede incluir TANTO bloques text como tool_use en una sola respuesta. El texto es comentario en lenguaje natural sobre lo que esta haciendo. El campo id del tool_use se usa para correlacionar con el tool_result posterior.

Valores de stop_reason — Tabla Completa

El loop agentico depende de stop_reason para decidir si continuar. Estos son los 6 valores posibles:

ValorSignificadoAccion
"end_turn"Claude produjo una respuesta finalSalir del loop
"tool_use"Claude quiere usar una o mas herramientasEjecutar tools, continuar loop
"max_tokens"La respuesta alcanzo el limite de max_tokensSalir del loop (respuesta truncada)
"stop_sequence"Se encontro una stop sequence configuradaSalir del loop
"refusal"Claude rechazo la solicitudSalir del loop (manejar rechazo)
"pause_turn"Loop server-side alcanzo limite de iteracionesRe-enviar conversacion para continuar

pause_turn y el loop server-side

Los server-executed tools (web_search, code_execution, etc.) ejecutan su propio loop dentro de la infraestructura de Anthropic. Si el modelo aun esta iterando cuando alcanza el limite de iteraciones, retorna stop_reason: "pause_turn". Esto indica que el trabajo NO esta terminado: debes re-enviar la conversacion completa (incluyendo la respuesta pausada) para que el modelo continue.

Regla Critica de Formato: Orden en tool_result

Los bloques tool_result deben ir PRIMERO en el array de content. El texto debe ir DESPUES de todos los tool results. Violar este orden causa un error 400.

INCORRECTO (causa error 400):

{ "role": "user", "content": [ { "type": "text", "text": "Here are the results:" }, { "type": "tool_result", "tool_use_id": "toolu_01" } ] }

CORRECTO:

{ "role": "user", "content": [ { "type": "tool_result", "tool_use_id": "toolu_01" }, { "type": "text", "text": "What should I do next?" } ] }

Ademas, los bloques tool_result deben seguir inmediatamente al mensaje del asistente con los tool_use correspondientes. No se pueden insertar mensajes entre ambos.

Hook Output — Campos de Retorno

Los hooks retornan un objeto con campos top-level y campos especificos en hookSpecificOutput:

  • systemMessage: inyecta un mensaje en la conversacion visible al modelo
  • continue (continue_ en Python): si el agente sigue ejecutando despues del hook

Campos de hookSpecificOutput para PreToolUse:

  • permissionDecision: "allow", "deny" o "ask"
  • permissionDecisionReason: string explicando la decision
  • updatedInput: input modificado del tool (requiere tambien permissionDecision: "allow")

Regla de Prioridad entre Hooks

deny > ask > allow — Si cualquier hook retorna "deny", la operacion se bloquea sin importar lo que retornen los demas hooks. Retornar {} (objeto vacio) equivale a permitir sin cambios.

Hook Output — Ejemplos Exactos

// Bloquear un tool return { systemMessage: "Remember: system directories like /etc are protected.", hookSpecificOutput: { hookEventName: "PreToolUse", permissionDecision: "deny", permissionDecisionReason: "Writing to /etc is not allowed" } }; // Modificar input (REQUIERE permissionDecision: "allow") return { hookSpecificOutput: { hookEventName: preInput.hook_event_name, permissionDecision: "allow", updatedInput: { ...toolInput, file_path: `/sandbox${originalPath}` } } };

Critico al usar updatedInput

Cuando usas updatedInput, DEBES incluir tambien permissionDecision: "allow". Siempre retorna un objeto nuevo en vez de mutar el tool_input original. El updatedInput debe estar dentro de hookSpecificOutput, no a nivel top.

Dominio 2 — 18% del examen

Tool Design & MCP Integration

Disenar interfaces de herramientas efectivas, implementar error handling estructurado, y configurar MCP servers.

2.1 Tool Descriptions Efectivas

Las descripciones son el mecanismo principal que los LLMs usan para seleccionar herramientas.

  • Descripciones minimas causan seleccion no confiable entre tools similares
  • Incluir formatos de input, queries de ejemplo, edge cases y limites
  • Renombrar tools para eliminar solapamiento funcional
  • Dividir tools genericos en tools especificos con contratos definidos

Ejemplo del examen

Si get_customer y lookup_order tienen descripciones minimas ("Retrieves customer info" / "Retrieves order details"), el primer paso es expandir las descripciones de cada tool con formatos de input, queries de ejemplo y limites — no agregar few-shot examples ni routing layers.

2.2 Error Responses Estructuradas

El patron isError de MCP para comunicar fallos al agente.

  • errorCategory: transient / validation / permission / business
  • isRetryable: boolean para prevenir reintentos inutiles
  • Distinguir access failures (timeout) de empty results validos (query exitosa sin matches)
  • Recovery local en subagentes; propagar solo errores irresolubles al coordinador
{ "isError": true, "content": [{ "type": "text", "text": JSON.stringify({ "errorCategory": "transient", "isRetryable": true, "description": "Database timeout after 30s", "attempted": "SELECT * FROM orders WHERE id=123", "partialResults": null }) }] }

2.3 Distribucion de Tools y tool_choice

Dar muchas tools a un agente (18 vs 4-5) degrada la confiabilidad de seleccion.

  • Restringir tools por rol: cada subagente solo recibe las herramientas de su especializacion
  • Tools cross-role limitadas para necesidades de alta frecuencia
  • tool_choice: "auto" — modelo puede retornar texto en vez de llamar tool
  • tool_choice: "any" — garantiza que el modelo llama alguna tool
  • tool_choice: {"type": "tool", "name": "..."} — fuerza tool especifico
tool_choiceComportamientoCaso de uso
"auto"Modelo decide si llamar tool o responderConversacion general
"any"Debe llamar alguna toolStructured output garantizado
{"type":"tool","name":"X"}Debe llamar tool XForzar extraction antes de enrichment

2.4 MCP Server Configuration

Model Context Protocol conecta agentes AI con sistemas externos.

  • Project scope: .mcp.json — shared via version control
  • User scope: ~/.claude.json — personal/experimental
  • Variables de entorno: ${GITHUB_TOKEN} en .mcp.json para credenciales
  • MCP resources exponen catalogos de contenido para reducir tool calls exploratorias
  • Preferir community MCP servers existentes sobre implementaciones custom
// .mcp.json - project-scoped { "mcpServers": { "github": { "command": "npx", "args": ["@modelcontextprotocol/server-github"], "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" } } } }

2.3+ strict: true — Sampling con Restriccion de Gramatica

Activar "strict": true como propiedad top-level en la definicion del tool (junto a name, description, input_schema) habilita grammar-constrained sampling.

  • Garantias: el input del tool sigue estrictamente el input_schema
  • El name del tool siempre es valido (de los tools provistos o server tools)
  • Las funciones reciben argumentos correctamente tipados cada vez — no necesitas validar y reintentar
  • Best practice: usar "additionalProperties": false en el input_schema con strict mode
  • Combinar tool_choice: {"type": "any"} + strict: true garantiza AMBAS cosas: que se llame un tool Y que los inputs cumplan el schema
  • Los schemas se compilan en gramaticas (mismo pipeline que structured outputs) y se cachean hasta 24 horas
  • HIPAA eligible, pero PHI no debe estar en las definiciones del schema (ni en nombres de propiedades, enum values, const values, ni pattern regexes)
{ "name": "get_weather", "description": "Get the current weather in a given location", "strict": true, "input_schema": { "type": "object", "properties": { "location": { "type": "string", "description": "The city and state, e.g. San Francisco, CA" }, "unit": { "type": "string", "enum": ["celsius", "fahrenheit"] } }, "required": ["location"], "additionalProperties": false } }

2.3+ tool_choice — Las 4 Opciones Completas

Cuatro opciones posibles para tool_choice, incluyendo "none":

OpcionSintaxisComportamiento
auto{"type": "auto"}Claude decide si llamar tools. Default cuando se proveen tools.
any{"type": "any"}Claude debe usar algun tool, pero no uno especifico
tool{"type": "tool", "name": "X"}Fuerza a Claude a usar siempre un tool particular
none{"type": "none"}Previene que Claude use cualquier tool. Default cuando no se proveen tools.

Restriccion con Extended Thinking

Con extended thinking habilitado, solo auto y none estan soportados. Usar any o tool retorna un error. Lo mismo aplica con Claude Mythos Preview (retorna error 400).

Detalle importante de prefill

Cuando tool_choice es any o tool, la API prefills el mensaje del asistente para forzar tool use. Esto significa que el modelo NO emitira lenguaje natural antes de los bloques tool_use, incluso si se le pide explicitamente. Cambiar tool_choice tambien invalida bloques de mensaje cacheados.

2.1+ Campo input_examples — Ejemplos Opcionales

Campo opcional en la definicion del tool para proveer ejemplos de inputs:

  • Array de objetos de ejemplo de input
  • Cada ejemplo debe ser valido segun el input_schema del tool (ejemplos invalidos retornan error 400)
  • No soportado para server-side tools — solo para tools user-defined y Anthropic-schema client tools
  • Costo en tokens: ~20-50 tokens para ejemplos simples, ~100-200 tokens para objetos nested complejos
  • Recomendado para tools con inputs complejos, objetos nested, o parametros sensibles al formato
{ "name": "get_weather", "description": "Get the current weather in a given location", "input_schema": { ... }, "input_examples": [ {"location": "San Francisco, CA", "unit": "fahrenheit"}, {"location": "Tokyo, Japan", "unit": "celsius"}, {"location": "New York, NY"} ] }

2.4+ Formatos .mcp.json — Configuraciones Exactas

Formatos exactos para las distintas configuraciones de MCP servers:

// Project scope (.mcp.json en raiz del proyecto) { "mcpServers": { "shared-server": { "command": "/path/to/server", "args": [], "env": {} } } } // HTTP server { "mcpServers": { "stripe": { "type": "http", "url": "https://mcp.stripe.com" } } } // Con OAuth { "mcpServers": { "my-server": { "type": "http", "url": "https://mcp.example.com/mcp", "oauth": { "clientId": "your-client-id", "callbackPort": 8080 } } } } // Con headersHelper (comando con timeout de 10s) { "mcpServers": { "internal-api": { "type": "http", "url": "https://mcp.internal.example.com", "headersHelper": "/opt/bin/get-mcp-auth-headers.sh" } } }

headersHelper

El comando debe escribir un objeto JSON de pares clave-valor a stdout. Se ejecuta con un timeout de 10 segundos. Los headers dinamicos sobreescriben headers estaticos con el mismo nombre. El helper se ejecuta fresco en cada conexion (sin cache).

2.4+ Las 3 Primitivas de MCP

MCP define tres primitivas fundamentales, cada una controlada por un actor diferente:

PrimitivaDescripcionQuien ControlaOperaciones
Tools Funciones que el LLM puede llamar activamente; pueden escribir en DBs, llamar APIs, modificar archivos Modelo (model-controlled) tools/list, tools/call
Resources Fuentes de datos pasivas que proveen acceso read-only a informacion para contexto Aplicacion (application-controlled) resources/list, resources/templates/list, resources/read, resources/subscribe
Prompts Templates de instrucciones pre-construidos que indican al modelo como trabajar con tools y resources especificos Usuario (user-controlled) prompts/list, prompts/get

Regla mnemotecnica para el examen

Tools = Modelo decide llamarlos | Resources = La aplicacion los gestiona | Prompts = El usuario los invoca (ej: slash commands, paletas de comandos, botones de UI).

2.4+ MCP — Precedencia de Scopes

Cuando multiples scopes definen el mismo servidor MCP, se aplica esta jerarquia de precedencia (de mayor a menor):

1
Local
Solo tu, solo este proyecto. Almacenado en ~/.claude.json (no en .claude/settings.local.json)
2
Project
Compartido via version control en .mcp.json del proyecto
3
User
Tu configuracion personal para todos los proyectos en ~/.claude.json
4
Plugin
Servers provistos por plugins instalados
5
Connectors
Connectors de claude.ai

Duplicados se resuelven por nombre (scopes) o por endpoint (plugins/connectors). CLI flags: --scope local (default), --scope project, --scope user.

2.4+ MCP — Nota sobre camelCase vs snake_case

Diferencia critica en el naming de schemas entre MCP y la Claude API:

Variables de entorno en .mcp.json

Sintaxis soportada: ${VAR} y ${VAR:-default}. Se expanden en command, args, env, url y headers. Si una variable requerida no esta definida y no tiene valor por defecto, Claude Code falla al parsear la config. Variables especiales de plugins: ${CLAUDE_PLUGIN_ROOT} y ${CLAUDE_PLUGIN_DATA}.

3.5 Iterative Refinement

  • Input/output examples son la forma mas efectiva de comunicar transformaciones
  • Test-driven iteration: escribir tests primero, iterar compartiendo failures
  • Interview pattern: hacer que Claude pregunte para surfacear consideraciones
  • Multiples issues interactivos en un mensaje vs sequencial para independientes

3.6 CI/CD Integration

  • -p / --print para modo no-interactivo en pipelines
  • --output-format json + --json-schema para output estructurado
  • CLAUDE.md provee contexto de proyecto en CI (standards, fixtures, review criteria)
  • Session context isolation: la misma sesion que genero codigo es menos efectiva revisandolo
# CI pipeline - review automatizado claude -p \ --output-format json \ --json-schema '{"type":"object","properties":{"findings":{"type":"array"}}}' \ "Review this PR for security issues. Output structured findings." # Incluir findings previos para evitar duplicados claude -p "Review changes since last commit. Previous findings: ${PREV_FINDINGS} Report only NEW or still-unaddressed issues."

Dominio 4 — 20% del examen

Prompt Engineering & Structured Output

Disenar prompts con criterios explicitos, few-shot examples, JSON schemas via tool_use, y batch processing.

4.1 Criterios Explicitos

  • Criterios especificos > instrucciones vagas ("flag comments only when behavior contradicts code" vs "check comments are accurate")
  • "Be conservative" o "only report high-confidence" NO mejora la precision
  • Categorias con alto false-positive rate destruyen confianza en categorias precisas
  • Deshabilitar temporalmente categorias problematicas mientras se mejoran prompts

4.2 Few-Shot Prompting

  • La tecnica mas efectiva para output consistente y formateado
  • 2-4 examples targeting ambiguous scenarios con reasoning
  • Muestran handling de casos ambiguos y razones de decision
  • Permiten generalizacion a patrones nuevos (no solo match literal)
  • Reducen hallucination en extraction tasks

Best Practice

Envolver ejemplos en tags <example> dentro de <examples>. Cada ejemplo debe mostrar input, output esperado y reasoning para decisiones ambiguas. 3-5 ejemplos diversos para mejores resultados.

4.3 Structured Output via tool_use

tool_use con JSON schemas es el approach mas confiable para output estructurado garantizado.

  • Elimina errores de sintaxis JSON, pero NO previene errores semanticos
  • Campos optional (nullable) cuando el source puede no tener la info → previene fabricacion
  • Enum con "other" + detail string para categorias extensibles
  • strict: true en tool definitions para conformancia de schema garantizada
# Extraction tool con JSON schema tools = [{ "name": "extract_invoice", "description": "Extract structured data from invoice", "input_schema": { "type": "object", "properties": { "vendor": {"type": "string"}, "total": {"type": "number"}, "date": {"type": ["string", "null"]}, # nullable! "category": { "type": "string", "enum": ["supplies", "services", "other"] }, "category_detail": {"type": ["string", "null"]} }, "required": ["vendor", "total"] } }] # Forzar que se use este tool tool_choice = {"type": "tool", "name": "extract_invoice"}

4.4 Validation & Retry Loops

  • Retry-with-error-feedback: incluir errores de validacion especificos en el retry prompt
  • Retries son inefectivos cuando la info simplemente no existe en el source
  • detected_pattern field para analizar patrones de falsos positivos
  • Self-correction: extraer calculated_total junto a stated_total para flaggear discrepancias

4.6 Multi-Instance Review

  • Self-review es limitado: el modelo retiene contexto de razonamiento previo
  • Instancia independiente (sin contexto previo) es mas efectiva para catch issues
  • Multi-pass: per-file local analysis + cross-file integration passes
  • Verification passes con confidence scores para routing calibrado

4.7 Adaptive Thinking

Reemplaza budget_tokens del extended thinking anterior. Claude decide dinamicamente cuanto pensar.

  • thinking: {"type": "adaptive"} reemplaza {"type": "enabled", "budget_tokens": N}
  • Calibrado por: parametro effort Y complejidad de la query
  • Niveles de effort: low, medium, high, max — max es exclusivo de Opus 4.6
  • En queries faciles, responde directamente sin pensar
  • Prefilled responses (ultimo turno asistente) deprecados en Claude 4.6
  • Migracion de prefill: usar Structured Outputs para formato, instrucciones directas para eliminar preambulos
client.messages.create( model="claude-opus-4-6", max_tokens=64000, thinking={"type": "adaptive"}, output_config={"effort": "high"}, messages=[...] )

Dominio 5 — 15% del examen

Context Management & Reliability

Preservar informacion critica, escalation patterns, error propagation, y human review workflows.

5.1 Context Window Management

  • Lost in the middle: modelos procesan bien inicio y final, pueden omitir secciones medias
  • Progressive summarization pierde numeros, fechas, porcentajes criticos
  • Extraer "case facts" transaccionales a un bloque persistente fuera del historial resumido
  • Trimming de tool outputs verbosos: mantener solo campos relevantes
  • Key findings al inicio + section headers explicitos para mitigar position effects

Patron: Case Facts Block

Extraer datos transaccionales (montos, IDs de orden, fechas, estados) en un bloque "case facts" que se incluye en cada prompt, separado del historial resumido. Esto previene perdida de datos criticos durante compaction.

5.2 Escalation & Ambiguity Resolution

  • Escalar cuando: cliente pide humano, policy gaps, no se puede progresar
  • Honrar request explicito del cliente para agente humano inmediatamente
  • Sentiment-based escalation y self-reported confidence NO son confiables
  • Multiples customer matches: pedir identificadores adicionales, no seleccion heuristica

Trampa comun del examen

Confidence scores self-reported (1-10) son poorly calibrated. El agente puede estar incorrectamente confiado en casos dificiles. Clasificadores de sentimiento resuelven un problema diferente (frustración =/= complejidad del caso).

5.3 Error Propagation Multi-Agent

  • Retornar contexto estructurado: tipo de fallo, query intentada, resultados parciales, alternativas
  • Anti-patron: suprimir errores silenciosamente (retornar vacio como exito)
  • Anti-patron: terminar workflow entero por un solo fallo
  • Subagentes: recovery local para errores transientes, propagar solo irresolubles

5.4 Large Codebase Exploration

  • Context degradation: respuestas inconsistentes refiriendo "typical patterns" en vez de clases descubiertas
  • Scratchpad files para persistir hallazgos clave entre limites de contexto
  • Subagent delegation para aislar output verboso de exploracion
  • /compact para reducir uso de contexto durante sesiones extendidas
  • Crash recovery: state exports estructurados que el coordinador carga al resumir

5.5 Human Review & Confidence Calibration

  • 97% accuracy global puede ocultar performance pobre en tipos especificos de documentos
  • Stratified random sampling de extracciones high-confidence para medir error rates
  • Field-level confidence scores calibrados con labeled validation sets
  • Validar accuracy por document type Y field antes de automatizar extracciones

5.6 Information Provenance

  • Source attribution se pierde durante pasos de summarization
  • Claim-source mappings estructurados que la sintesis debe preservar
  • Datos conflictivos: anotar con source attribution, no seleccionar uno arbitrariamente
  • Temporal data: requerir publication/collection dates para interpretacion correcta
  • Secciones explicitas distinguiendo hallazgos bien-establecidos de contestados

5.7 Context Window Startup

5.8 Hooks en settings.json

Sistema de hooks para interceptar y controlar acciones de Claude Code.

Tipo de HandlerDescripcion
commandEjecuta un script/comando. Soporta async, asyncRewake, shell, timeout, once
httpLlama a un endpoint HTTP. Soporta headers, allowedEnvVars, timeout
promptConsulta a un modelo con un prompt. Soporta model, timeout
agentLanza un agente para verificar/procesar. Soporta prompt, timeout
  • Exit code 2 bloquea la accion (en PreToolUse, PermissionRequest, UserPromptSubmit, Stop, etc.)
  • Output cap: 10,000 caracteres para additionalContext, systemMessage, updatedMCPToolOutput

Ventana de contexto: 200,000 tokens. Skills despues de compaction: 5,000 tokens por skill, 25,000 tokens de presupuesto total combinado. Los skills invocados mas recientemente tienen prioridad. Skills con disable-model-invocation: true cuestan cero contexto hasta ser invocados manualmente.
